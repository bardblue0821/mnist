# 要求仕様書

## 概要

- 手書き数字の画像を入力として受け取り、その数字を認識するシステムを開発する。
- Web アプリケーション。React+Next.js（静的エクスポート）＋Python(FastAPI)
- CSS フレームワーク: **Tailwind CSS**
- UI コンポーネント: **shadcn/ui**（Button, Card, Progress, Skeleton）
- ユーザーはブラウザ上で手書き数字を描画し、送信することで認識結果を得ることができる。
- ０～９のうち、どの数字が描かれているかを％で棒グラフ表示する。
- 入力データはデータベースに保存しない。
- UI の言語は日本語のみとする。
- TDD（テスト駆動開発）で開発する。
- monorepo 構成（`frontend/` / `backend/`）で 1 リポジトリに管理する。

## 機能要件

### 1. ユーザーインターフェース
- ユーザーはブラウザ上で手書き数字を描画できるキャンバスを提供する。
  - キャンバスサイズ: 280×280px（MNIST の 28×28 の 10 倍）
  - 背景色: 黒、ペン色: 白、ペンの太さ: 固定（太め、約 15px）
  - MNIST データセット（黒背景に白文字）と同じ配色のため、色反転処理は不要
  - マウスクリック＆ドラッグで描画（タッチ操作にも対応）
- 描画をクリアするための「クリア」ボタンを設置する。
- **リアルタイム推論**: 描画が 500ms 停止したら自動で判定を実行する（デバウンス）。
  - 「判定」ボタンは設置しない。描画中に棒グラフがリアルタイムで更新される体験を提供する。
- 認識結果を 0〜9 の全数字について棒グラフで表示する。
  - 各数字の確率（％）を水平バーで視覚的に表示する。
  - 最も確率の高い数字をハイライト表示する。
  - 確率値の変化時に CSS トランジションでバーを滑らかにアニメーションさせる。
- エラー発生時は **sonner**（shadcn/ui 推奨）によるトースト通知で表示する。
- レスポンシブデザインを採用し、PC・スマートフォン・タブレットで利用可能にする。

### 2. バックエンド
- FastAPI を使用して、フロントエンドからのリクエストを処理する。
- フロントエンドとバックエンドは同一ドメインで運用する（CORS 設定不要）。
- API エンドポイント:
  - `POST /api/predict` — 数字認識
    - リクエスト: `{ "image": "<Base64エンコードされたPNG画像>" }`
    - レスポンス: `{ "predictions": [{"digit": 0, "probability": 0.05}, ..., {"digit": 9, "probability": 0.01}] }`
    - エラーレスポンス: `{ "detail": "<エラーメッセージ>" }` (HTTP 400/500)
  - `GET /api/health` — ヘルスチェック
    - レスポンス: `{ "status": "ok" }`
- 受け取った Base64 PNG 画像を前処理する:
  - PNG デコード → グレースケール変換 → 28×28px にリサイズ → 正規化（0〜1）
  - キャンバスが MNIST と同じ配色（黒背景・白文字）のため、色反転は不要
- 学習済みモデルを使用して推論し、0〜9 各数字の確率を返す。
- エラーハンドリングを実装し、無効な入力やサーバーエラーに対して適切なレスポンスを返す。
- 入力データのバリデーション（Base64 形式チェック、画像サイズ上限など）を行う。
- ログ機能を実装し、リクエスト・エラーを記録する。

### 3. 機械学習モデル
- MNIST データセットを使用して、手書き数字認識モデルを構築する。
- モデルアーキテクチャ: **CNN（畳み込みニューラルネットワーク）**
- フレームワーク: **TensorFlow / Keras** でトレーニングを行う。
- モデルの保存形式: **TFLite 形式**（`.tflite`）
  - Keras で学習 → TFLite に変換して保存 → バックエンドで `tflite-runtime` を使用して推論
  - TFLite はモデルサイズが小さく、メモリ消費が少ないため Render 無料枠（512MB）に適する
- モデルはアプリケーション起動時に一度ロードし、メモリ上に保持する（リクエストごとのロードを避ける）
- **精度目標: テストデータで 95% 以上の正解率を達成すること。**

### 4. デプロイメント
- アプリケーションをクラウドサービス **Render** にデプロイする。
  - FastAPI で Next.js の静的ビルド成果物もホストし、単一の Docker コンテナで運用する。
  - これにより同一ドメインでフロント・バックエンドを提供する。
- Dockerfile を作成し、再現性のあるビルドを実現する。
- デプロイ後の動作確認を行い、必要に応じて修正を加える。
- デプロイメントの手順をドキュメント化し、再現性を確保する。

## 非機能要件

### 1. パフォーマンス
- システムは、ユーザーが数字を描画してから認識結果が表示されるまでの時間を 2 秒以内にすること。
- フロントエンドは、ユーザーの操作に対して迅速に反応すること。

### 2. セキュリティ
- 入力データのバリデーションを行い、不正なリクエストからシステムを保護すること。
- デプロイメント環境において、セキュリティパッチを適用し、脆弱性を最小限に抑えること。
- 同一ドメイン運用により、CORS に起因するセキュリティリスクを回避する。

### 3. 対応環境
- PC: Google Chrome（最新版）、Microsoft Edge（最新版）
- スマートフォン: iOS Safari（最新版）、Android Chrome（最新版）

### 4. 可用性
- 個人プロジェクトのため、高い可用性は求めない。
- Render 無料枠のスリープ仕様（一定時間アクセスがないとスリープ）を許容する。
- コールドスタート時のウォームアップ表示: 初回アクセス時に `/api/health` でサーバー状態を確認し、起動中の場合は「サーバー起動中...」を表示する。

### 5. テスト
- TDD（テスト駆動開発）で開発する。
- フロントエンド: Jest + React Testing Library によるユニットテスト。
- バックエンド: pytest によるユニットテスト・API テスト。
- モデル: トレーニングスクリプトに評価テスト（精度 95% 以上の検証）を含める。
